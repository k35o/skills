# 書き方パターン

## 出力フォーマットの定義

**厳密な要件の場合:**

```markdown
## レポート構成

必ずこのテンプレートに従う:

# [タイトル]

## 概要
[主要な発見の要約]

## 詳細
- 項目1: データ付きの説明
- 項目2: データ付きの説明

## 推奨事項
1. 具体的なアクション
2. 具体的なアクション
```

**柔軟なガイドの場合:**

```markdown
## レポート構成

以下をデフォルトとし、状況に応じて調整する:

# [タイトル]
## 概要
## 詳細（内容に応じてセクションを調整）
## 推奨事項
```

## 例の提示

出力の品質が例に依存する場合、入出力ペアを示す:

```markdown
## コミットメッセージの形式

**例1:**
入力: JWTトークンによるユーザー認証を追加
出力: feat(auth): JWTベースの認証を実装

**例2:**
入力: レポートで日付が正しく表示されないバグを修正
出力: fix(reports): タイムゾーン変換の日付フォーマットを修正
```

説明だけよりも、例の方がスタイルや粒度を正確に伝えられる。

## ワークフローの定義

**逐次実行:**

```markdown
PDFフォームの入力手順:

1. フォームを解析する（analyze_form.pyを実行）
2. フィールドマッピングを作成する（fields.jsonを編集）
3. マッピングを検証する（validate_fields.pyを実行）
4. フォームに入力する（fill_form.pyを実行）
```

**条件分岐:**

```markdown
1. 変更の種類を判断する:
   **新規作成の場合** → 「作成ワークフロー」へ
   **既存の編集の場合** → 「編集ワークフロー」へ

2. 作成ワークフロー: [手順]
3. 編集ワークフロー: [手順]
```

## 自由度の設定

タスクの性質に応じて指示の具体性を調整する:

**高い自由度**（複数のアプローチが有効）: テキストベースの指示でヒューリスティクスを示す

**中程度の自由度**（推奨パターンはあるが変動も許容）: 擬似コードやパラメータ付きスクリプト

**低い自由度**（操作が壊れやすく一貫性が重要）: 具体的なスクリプト、少ないパラメータ
