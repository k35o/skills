# テストダブルの使い方

## 分類

| 種類 | 役割 | 例 |
|------|------|-----|
| Stub | 固定値を返す | APIレスポンスの固定化 |
| Spy | 呼び出しを記録する | 関数が正しい引数で呼ばれたか確認 |
| Mock | 期待する呼び出しを事前定義する | 特定の順序・引数での呼び出しを検証 |
| Fake | 簡易的な代替実装 | インメモリDB、LocalStack |

## 使うべきとき

- 外部サービスへの依存を切り離すとき
- テストの実行速度を上げたいとき
- 決定的（deterministic）な結果が必要なとき
- テスト対象以外の障害でテストが失敗するのを防ぎたいとき

## 使いすぎの危険

テストダブルの多用は**構造的結合**を生む。プロダクトコードの内部構造が変わるとテストが壊れる。

```tsx
// ❌ 実装の内部構造に結合
const mockRepo = jest.fn().mockReturnValue(user);
service.getUser(1);
expect(mockRepo).toHaveBeenCalledWith(1);
// → repositoryのメソッド名やシグネチャが変わると壊れる

// ✅ 振る舞いで検証
const result = service.getUser(1);
expect(result).toEqual({ id: 1, name: 'Alice' });
```

## Fakeの推奨

可能であればMock/StubよりもFakeを使う。Fakeは代替実装なので、プロダクトコードとの構造的結合が少ない。

例:
- DynamoDB → DynamoDB Local（コンテナ）
- 外部API → インメモリの簡易実装
- ファイルシステム → インメモリFS

Fakeを使うことでテストサイズをLargeからMediumに下げられる。

## テストダブルと自作自演

テストダブルで「呼ばれたこと」だけを検証すると、自作自演テストになりやすい。テストダブルを使っても、最終的な振る舞い（出力、副作用の結果）を検証する。

## 原則

- テストダブルはテスト対象の**境界**に置く
- テスト対象の**内部**にテストダブルを挟まない
- できるだけFakeを使い、Mock/Stubは最小限に
- テストダブルを使うことで「何をテストしているか」が曖昧にならないようにする
